# GitHub App Live Smoke Runbook

## Purpose

Issue #4 cannot be closed until the GitHub App flow is verified with real credentials. This runbook defines the minimum live smoke procedure for connect, publish, reconcile, marker search, and controlled retry without storing secrets in the repository.

## Scope

- GitHub App installation connection
- GitHub Issue publish from an approved Issue Draft
- Publish attempt audit trail
- Marker search reconciliation
- Manual link
- Controlled retry approval

Out of scope:

- GitHub webhook delivery verification
- permission change synchronization
- production incident response
- full accessibility review

## Secret Handling

Never paste real credentials into Git, docs, screenshots, logs, or AI chat.

Use environment variables or a deployment secret store:

```sh
GITHUB_ISSUE_PUBLISH_PROVIDER=github_app
GITHUB_APP_ID=123456
GITHUB_APP_SLUG=ai-pm-platform
GITHUB_APP_PRIVATE_KEY_BASE64=base64-encoded-pem
GITHUB_APP_INSTALLATION_URL=https://github.com/apps/ai-pm-platform/installations/new
GITHUB_API_BASE_URL=https://api.github.com
GITHUB_WEBHOOK_SECRET=webhook-secret-from-secret-store
```

Prefer `GITHUB_APP_PRIVATE_KEY_BASE64` over raw PEM in shell environments to avoid newline corruption. Do not set both private key variables unless intentionally testing precedence.

Required GitHub App permissions:

- Metadata: read
- Issues: write

## GitHub Webhook secret rotation

関連ADR: `docs/decisions/ADR-0021_github_webhook_secret_rotation.md`

webhook secret、signature、payload全文、GitHub delivery id生値は、このrepository、screenshot、review docs、log、AI chatへ保存しない。delivery証跡はdelivery digestとsafe metadataだけを使う。

### 通常rotation

1. 新しいwebhook secretをsecret storeへ登録する。
2. appを以下でdeployする。

```sh
GITHUB_WEBHOOK_SECRET=new-secret
GITHUB_WEBHOOK_PREVIOUS_SECRET=old-secret
```

3. GitHub App側のwebhook secretを新しい値へ更新する。
4. 新secretで署名したstaging test deliveryを実行する。
5. rotation window中にprevious secret署名のdeliveryも受理されることを確認する。
6. 24時間以内に `GITHUB_WEBHOOK_PREVIOUS_SECRET` を削除する。
7. `GITHUB_WEBHOOK_SECRET` のみでappをdeployする。
8. secret値をredactした証跡を `docs/review/` へ保存する。

### 緊急rotation

旧secretの漏えいが疑われる場合に使う。

1. `GITHUB_WEBHOOK_PREVIOUS_SECRET` を設定しない。
2. GitHub App側のwebhook secretを即時変更する。
3. `GITHUB_WEBHOOK_SECRET=new-secret` のみでappをdeployする。
4. 旧secret署名deliveryが拒否されることを確認する。
5. incident理由、影響、operator、検証結果をreview documentへ記録する。

### Rollback

新secretで正当なdeliveryを検証できない場合:

1. GitHub App側のwebhook secretを旧値へ戻す。
2. `GITHUB_WEBHOOK_SECRET=old-secret` でappをdeployする。
3. `GITHUB_WEBHOOK_PREVIOUS_SECRET` を削除する。
4. signed deliveryの受理を確認する。
5. rollback理由、operator、開始/終了時刻、commit SHA、safe failure detailを保存する。

Rotation evidenceには以下を含める:

- 対象environment
- commit SHA
- operator / approver
- rotation理由
- previous secret設定時刻
- previous secret削除期限
- previous secret削除時刻
- current secret署名deliveryの結果
- rotation window中のprevious secret署名delivery結果
- raw secret、signature、payloadを保存していない確認

## Preconditions

- GitHub App exists.
- GitHub App is installed on the target repository.
- The target repository is disposable or approved for smoke testing.
- Test project has `github_repo` set to the target `owner/repo`.
- App server can reach `https://api.github.com`.
- `GITHUB_ISSUE_PUBLISH_PROVIDER=github_app` is set.
- `integration_accounts` can be created through the connection callback flow.
- CI is green on the commit under test.

## Test Data

Use a clearly disposable Issue Draft title:

```text
[AI PM Smoke] GitHub App publish and reconciliation check
```

The Issue body must include the AI PM marker generated by the app. Do not manually remove markers during smoke testing.

## Smoke Steps

### 1. Connection Start

1. Open the workspace UI.
2. Select or create a Project with the target GitHub repository.
3. Start GitHub connection.
4. Confirm the generated URL points to the configured GitHub App installation URL.
5. Complete installation or repository selection in GitHub.
6. Return through the callback.

Expected:

- `integration_accounts` has one active GitHub connection for the Project and repository.
- No raw private key, JWT, installation token, or state token appears in logs.
- AuditLog records connection start/callback/disconnect events where implemented.

### 2. Publish

1. Create or select an approved Requirement.
2. Generate an Issue Draft.
3. Ensure the latest OpenAPI Draft is `valid` or `approved`.
4. Approve the Issue Draft.
5. Publish to GitHub.

Expected:

- GitHub Issue is created in the target repository.
- App stores `github_issue_number`, `github_issue_url`, repository, and attempt id.
- `github_issue_publish_attempts` includes publish started and local save completed states.
- `issue_draft.github_published` AuditLog includes `attempt_id`.
- Idempotency digest is stored, but the raw Idempotency-Key is not exposed.

### 3. Marker Search Reconciliation

Use this only if publish returns reconciliation required or if a controlled test setup intentionally simulates local save failure after GitHub creation.

1. Run marker search reconciliation from the UI.
2. Confirm candidate count and metadata.
3. If exactly one correct candidate is returned and search is complete, allow automatic reconcile.
4. If zero or multiple candidates are returned, keep the Review blocker open and resolve manually.

Expected:

- Search metadata includes total count and incomplete result status.
- `incomplete_results=true` never auto-reconciles.
- Candidate title, state, updated time, score, repository, and URL are visible.
- AuditLog records reconciliation result.

### 4. Manual Link

1. Enter GitHub Issue number.
2. Enter matching GitHub Issue URL from the same repository.
3. Add a resolution note.
4. Link existing Issue.

Expected:

- URL repository and Issue number mismatch is rejected before or by the API.
- Successful link sets the Issue Draft to published.
- Review blocker is resolved with a resolution note.
- AuditLog records the manual reconciliation.

### 5. Controlled Retry

Use controlled retry only after confirming the GitHub Issue was not created.

1. Confirm marker search found no matching Issue and search was complete.
2. Enter resolution note.
3. Enter retry approver.
4. Select retry reason template.
5. Approve retry.

Expected:

- Missing approver or reason template is rejected.
- Retry approval stores approver and reason in attempt detail, Review resolution note, and AuditLog metadata.
- Issue Draft returns to approved status.
- Cooldown blocks retry approval until `next_reconciliation_retry_at` has passed.

## Failure Handling

Stop the smoke and record the result if any of the following occurs:

- GitHub App credential is missing or invalid.
- GitHub returns permission denied.
- The repository in callback does not match the Project repository.
- A raw token or private key appears in logs.
- Reconciliation is required but marker search cannot confirm safe state.
- Controlled retry is requested without a human approver and reason.

Do not manually create or delete GitHub Issues as a workaround without recording the operator action in the related Review or Issue.

## Evidence To Save

Save the following in the related review:

- evaluation datetime
- commit SHA
- environment name
- GitHub repository
- GitHub Issue URL created by smoke
- related `issue_draft.id`
- related `github_issue_publish_attempt.id`
- AuditLog actions observed
- screenshots with secrets redacted
- pass/fail conclusion

## Completion Criteria

The live smoke is complete only when:

- connect succeeds
- publish succeeds
- attempt and AuditLog are present
- marker search can locate the AI PM marker
- manual link or controlled retry path is verified in a safe test scenario
- no secret leakage is observed
- review is saved under `docs/review/`
