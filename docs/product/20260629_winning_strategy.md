# 2026-06-29 勝ち筋

## ポジショニング

AI議事録ツールではなく、AI PM Platform として位置づける。

キャッチコピー案:

```text
会議を、実装可能なプロジェクト計画へ変える。
```

## 競争優位

### 1. ReviewOpsをプロダクトの中核にする

既存AI議事録は「要約して終わり」になりやすい。本プロダクトは、各工程でレビューを必須化し、改善履歴を成果物として残す。

### 2. GitHub-first

開発チームが実際に仕事を進める単位はIssue、Pull Request、レビュー、CIである。議事録ではなく、GitHub上の実行単位へ変換する。

### 3. API-first

要件からOpenAPIを生成し、レビュー後にBackend/Frontendへ進む。AI実装の失敗原因である曖昧な仕様を減らす。

### 4. Discord-first

SlackやTeamsよりも軽量な開発コミュニティ、個人開発、AIエージェントチームに刺さる。最初の市場を狭く取り、深く刺す。

### 5. AI実装エージェントの前工程を押さえる

Codexなどの実装AIの価値は、入力仕様の品質に依存する。本プロダクトは、AIが実装しやすい仕様、Issue、API、レビューを作る。

## MVPの勝ち筋

MVPでは全自動PMを目指さない。まず、以下の狭いが強い価値を提供する。

```text
Discord会議ログまたは音声/文字起こし
-> AI議事録
-> 要件定義ドラフト
-> GitHub Issueドラフト
-> OpenAPIドラフト
-> 専門家レビュー
-> 改善アクション
```

## やらないこと

- 汎用議事録ツールとしての全会議アプリ対応
- 高度なカレンダー運用
- Jira/Asana/Linearへの同時対応
- Slack正式対応
- 完全自動マージ
- AI判断だけでのリリース承認

## 評価指標

- 会議からIssue化までの時間
- 人間が修正したIssue本文の割合
- AI生成Issueから実装完了した割合
- OpenAPIレビューでの重大指摘数
- 手戻りIssue数
- レビュー未実施フェーズ数
- 週次アクション完了率
